# TUI ファイルマネージャ レイアウト調査

## 概要表

| ツール | 言語 | レイアウト種別 | デフォルトペイン数 | 比率 | マウス | タブ |
| --- | --- | --- | --- | --- | --- | --- |
| yazi | Rust | Miller columns (3列) | 3 | 1:4:3 | 対応 | 対応 |
| ranger | Python | Miller columns (3列) | 3 | 1:2:3 (概算) | 対応 | 対応 |
| lf | Go | 設定可能な列数 | 3 | 1:2:3 | 任意 | 非対応 |
| nnn | C | 単一列 | 1 | N/A | 非対応 | Context (4) |
| broot | Rust | Tree + 任意パネル | 1-2 | 動的 | 対応 | 非対応 |
| superfile | Go | Sidebar + 複数パネル | 2-4 | 動的 | 対応 | 非対応 |

## 詳細分析

### 1. yazi

#### yazi のレイアウト構造

3列の Miller columns レイアウト:

```txt
+--------+------------------+--------------+
| Parent | Current          | Preview      |
| (1)    | (4)              | (3)          |
+--------+------------------+--------------+
```

デフォルト比率は `1:4:3`。
現在のディレクトリ列が最も広く、
次にプレビューペインが広い。

`yazi.toml` で完全にカスタマイズ可能:
`ratio = [1, 4, 3]`。列数と相対幅を変更できる。

#### yazi のナビゲーション

- Vi ライクなキーバインド (h/j/k/l)
- マウス対応: click と scroll イベント
  (`mouse_events = ["click", "scroll"]`)
- 矢印キーも併用可能
- rename, filter, search 等のモーダル入力

#### yazi のタブ / ワークスペース

- 名前付きのマルチタブ対応
- タブ間でのディレクトリ横断選択
- 画面上部にタブバー表示

#### yazi の検索 / フィルタ

- インクリメンタル filter-as-you-type
- 外部ツール連携: ripgrep, fd, fzf
- プラグインによるファジー検索

#### yazi のステータスバー

- ヘッダー行: タブバーと現在のパス
- フッター行: ファイルメタデータ (権限, サイズ, 日時)
- フッター領域にコンテキスト情報

#### yazi のプレビューシステム

- treesitter によるコードハイライト
- 複数ターミナルプロトコルで画像プレビュー
  (Kitty, iTerm2, Sixel)
- コード、PDF、アーカイブのスクロール可能なプレビュー
- プレビューの非同期プリロード

#### yazi のレスポンシブデザイン

- Lua プラグインで UI 全体の書き換えが可能
- 比率ベースのレイアウトがターミナル幅に適応
- プラグインによるカスタムレイアウト

### 2. ranger

#### ranger のレイアウト構造

TUI ファイルマネージャにおける Miller columns の
先駆者。3列レイアウト:

```txt
+--------+-----------+------------------+
| Parent | Current   | Preview          |
| (1)    | (2)       | (3)              |
+--------+-----------+------------------+
```

2つ以上の等幅ペインで異なるディレクトリを並べる
"mc-style" viewmode もサポート。

#### ranger のナビゲーション

- Vi ライクなキーバインド (h/j/k/l)
- 矢印キーでのナビゲーション
- ファイルクリックのマウス対応
- Page up/down, home/end

#### ranger のタブ / ワークスペース

- 番号付きタブ対応
- ブックマークによるディレクトリ高速アクセス

#### ranger の検索 / フィルタ

- `/` で現在ディレクトリ内の正規表現検索
- `f` で名前によるクイック検索
- マッチするファイルのみ表示するフィルタモード

#### ranger のステータスバー

- タイトルバー: ホスト名、現在のパス
- ステータスバー (下部): ファイル情報、権限、
  オーナー、サイズ、更新日時
- バックグラウンド操作のタスクビュー

#### ranger のプレビューシステム

- カスタマイズ可能な `scope.sh` スクリプト
- シンタックスハイライト用外部ツール
  (highlight, bat, pygmentize)
- w3mimgdisplay, ueberzug, kitty protocol
  による画像プレビュー
- PDF、アーカイブ、動画サムネイル対応

#### ranger のレスポンシブデザイン

- 列幅が比例的に調整
- 狭いターミナルで graceful に縮退

### 3. lf

#### lf のレイアウト構造

設定可能な複数列レイアウト:

```txt
+--------+-----------+------------------+
| Parent | Current   | Preview          |
| (1)    | (2)       | (3)              |
+--------+-----------+------------------+
```

デフォルト比率は `1:2:3`。
`ratios` 設定の項目数でペイン数が決まる。
単一値にすると single-pane モードになる。
`drawbox` オプションで罫線文字を使用可能。

`ratios` は Miller column ファイルマネージャの中で
最も柔軟:

- `set ratios 1:2` で2ペイン (プレビューなし)
- `set ratios 1:2:3` で3ペイン (デフォルト)
- `set ratios 1:2:3:5` で4ペイン

リモートコマンドでターミナル幅に応じた
動的再設定が可能。

#### lf のナビゲーション

- Vi ライクなキーバインド (h/j/k/l)
- マウス対応 (任意, `set mouse`)
  click, right-click, scroll
- ビジュアルモード (V) で選択

#### lf のタブ / ワークスペース

- タブ機能なし。
  tmux/screen 等のターミナルマルチプレクサに委譲
- サーバー/クライアントアーキテクチャで
  複数インスタンスの状態共有が可能

#### lf の検索 / フィルタ

- `/` で前方検索、`?` で後方検索
- `f` で文字ベースのジャンプ
- text, glob, regex によるフィルタモード
- インクリメンタル検索とフィルタオプション

#### lf のステータスバー

- 上部: user@host:path 形式のプロンプト行
  (`promptfmt` でカスタマイズ可能)
- 下部: ファイル統計情報 (権限, ユーザー,
  グループ, サイズ, 日時) とステータスインジケータ
  (選択数, copy/cut 表示, スクロール位置)
- Go テンプレートで完全にカスタマイズ可能

#### lf のプレビューシステム

- 外部プレビュースクリプト (ranger の scope.sh 同様)
- プレビュー後のクリーンアップスクリプト
- プレビューのキャッシュとプリロード
- ディレクトリプレビューのオプション (`dirpreviews`)

#### lf のレスポンシブデザイン

- 比率ベースのレイアウトが自動適応
- `on-redraw` やリモートコマンドで
  ターミナル幅に応じた比率再設定が可能
- 例: 80列以下で2列、160列以下で3列、
  160列超で4列

### 4. nnn

#### nnn のレイアウト構造

デフォルトは単一列:

```txt
+----------------------------+
| Current Directory Listing  |
|                            |
|                            |
|                            |
+----------------------------+
| Status / Info Line         |
+----------------------------+
```

単一ディレクトリビューに集中したミニマルな設計。
"Detail" モードでファイル情報 (権限, オーナー,
サイズ, 日時) を各エントリ横に表示。

非常に小さいターミナル用の "1-column mode" 対応。

#### nnn のナビゲーション

- 矢印キー、vim キー、カスタムショートカット
- `Tab` でコンテキストの作成と切り替え
- "Type-to-nav" モードで即時フィルタリング
- 相対オフセットによるエントリジャンプ
- 徹底したキーボード中心設計

#### nnn のタブ / ワークスペース

- タブではなく4つの "context" (ワークスペース)
- 各コンテキストが独自のディレクトリ状態を保持
- コンテキストごとのカスタムカラー
- セッション永続化対応

#### nnn の検索 / フィルタ

- リアルタイム filter-as-you-type
- string, POSIX regex, PCRE2 regex 対応
- ユニークマッチ時の自動ディレクトリ移動
- プラグインによるサブツリー検索

#### nnn のステータスバー

- 下部行: コンテキストインジケータ、
  ディレクトリ情報、ファイル数
- Detail モードで列追加: 権限, オーナー,
  サイズ, 更新日時
- デフォルトはミニマル

#### nnn のプレビューシステム

- 組み込みのプレビューペインなし
- FIFO ベースのプレビュー (プラグイン) が
  別ターミナルペインで開く
- tmux split 等を使った外部プラグインで
  "ライブプレビュー"

#### nnn のレスポンシブデザイン

- 単一列が自然にあらゆる幅に適応
- 30列未満のターミナルで 1-column モード
- 極めて低いメモリ使用量 (<3.5MB)

### 5. broot

#### broot のレイアウト構造

検索統合型の独特なツリーベースレイアウト:

```txt
+----------------------------+
| / (root)                   |
|   src/                     |
|     main.rs                |
|     lib.rs                 |
|   Cargo.toml               |
| ...                        |
+----------------------------+
| Search / Command Input     |
+----------------------------+
```

プレビューや2番目のパネルを開いた場合:

```txt
+--------------+--------------+
| Tree View    | Preview/     |
|              | Second Tree  |
|              |              |
+--------------+--------------+
| Input Line                  |
+-----------------------------+
```

ツリービューは剪定された関連性フィルタ付き
ディレクトリツリーを表示。
"Unlisted" カウントで非表示エントリ数を表示。

#### broot のナビゲーション

- 検索駆動: タイプすると即座にツリーがフィルタ
- 矢印キーで選択移動
- Enter でディレクトリフォーカスまたはファイル開く
- Alt+Enter で cd して終了
- Ctrl+Left/Right でパネル間移動

#### broot のタブ / ワークスペース

- タブなし。代わりにパネルを使用
- Ctrl+Right で選択ディレクトリまたはファイル
  プレビューの新規パネルを開く
- 複数パネルを横に並べて表示可能

#### broot の検索 / フィルタ

- 中核的な革新: ファジー検索がツリービューに統合
- マッチしないブランチはすべて剪定される
- `/` プレフィックスで正規表現検索
- `c/` プレフィックスでコンテンツ検索
- 論理演算子 (`&`, `|`, `!`) でパターン結合

#### broot のステータスバー

- 下部エリア: 検索/コマンド用の入力行
- Git ブランチとステータスインジケータ (利用可能時)
- ファイル情報はコンテキストに応じて表示

#### broot のプレビューシステム

- Ctrl+Right でファイルのプレビューパネルを開く
- シンタックスハイライト付きコードプレビュー
- 画像プレビュー (Kitty graphics protocol)
- プレビューがツリー選択と同期

#### broot のレスポンシブデザイン

- ツリー深度がターミナル高さに自動調整
- パネル幅が均等または比例的に分割
- 小さいターミナルではツリーレベル数が減少

### 6. superfile

#### superfile のレイアウト構造

サイドバーとフッターエリアを持つ
マルチパネルレイアウト:

```txt
+---------+----------+----------+
| Sidebar | Panel 1  | Panel 2  |
| (pinned |          |          |
|  dirs,  |          |          |
|  disks) |          |          |
|         |          |          |
+---------+----------+----------+
| Processes | Metadata | Clipboard|
+----------+----------+----------+
```

主要コンポーネント:

- Sidebar: ピン留めディレクトリとディスク情報
- ファイルパネル: 1-3パネルでディレクトリ表示
- フッターパネル: プロセスバー、メタデータ、
  クリップボード
- オプションのファイルプレビューパネル

各コンポーネントが独自のボーダー付きレンダラーを持ち、
フォーカス時にボーダー色が変化。

#### superfile のナビゲーション

- Vim スタイルまたはデフォルトのホットキープリセット
- h/j/k/l または矢印キー
- マウス対応
- sidebar, panels, footer 間のフォーカス切り替え

#### superfile のタブ / ワークスペース

- タブなし。複数パネルが同等の役割を果たす
- パネルの動的な作成/削除
- 各パネルが独立したディレクトリ状態を保持

#### superfile の検索 / フィルタ

- 現在のパネル内での検索
- 拡張検索用のプラグインシステム

#### superfile のステータスバー

- 従来型のステータスバーなし
- フッターパネルがコンテキスト情報を提供:
  - プロセスバー: ファイル操作の進捗
  - メタデータ: 選択ファイルの詳細
  - クリップボード: copy/cut バッファ

#### superfile のプレビューシステム

- 別プレビューパネル (任意, toggle on/off)
- プレビュー周囲のボーダー設定可能
  (`EnableFilePreviewBorder`)

#### superfile のレスポンシブデザイン

- パネルが利用可能な幅を均等に分割
- Sidebar は固定幅または比例幅
- フッターパネルは下部エリアを均等分割
- ターミナル幅に応じてパネル数を調整

## 主要デザインパターン

### パターン 1: Miller Columns (yazi, ranger, lf)

Parent-Current-Preview の3列レイアウトが主流パターン。

主な特徴:

- 空間的コンテキストの提供: どこから来たか (parent)、
  今どこにいるか (current)、
  何を見ているか (preview)
- ディレクトリ階層ナビゲーションに自然にマッピング
- 比率ベースのサイズ調整で最重要列を強調可能
- h/l のナビゲーションがディレクトリツリー上の
  スライディングウィンドウのように感じられる

コードレビューでの強み:

- プレビューペインがファイル内容表示に自然にマッピング
- Parent 列がプロジェクト構造のコンテキストを提供
- vim/ranger ユーザーに馴染みがある

コードレビューでの弱み:

- 特定ファイルのレビュー時に Parent 列の有用性が低い
- コードの長い行に対してプレビュー幅が制限的

### パターン 2: 単一列 + 外部プレビュー (nnn)

リスト領域を最大化し、プレビューを外部ツールに委譲。

コードレビューでの強み:

- ファイルリストに最大の画面領域
- クリーンで集中しやすいインターフェース

コードレビューでの弱み:

- インラインプレビューなし。ウィンドウ管理が必要
- リストとプレビュー間のコンテキスト切り替え

### パターン 3: Tree + 検索駆動 (broot)

ナビゲーションと検索が統合。
ツリーが構造的コンテキストを提供し、
検索フィルタが可視セットを絞り込む。

コードレビューでの強み:

- 大規模コードベースでの特定ファイル検索に優秀
- ツリー構造でプロジェクト階層を一目で把握
- search-as-you-type が非常に効率的

コードレビューでの弱み:

- 順次ファイルレビューにはやや不向き
- プレビューがナビゲーションに対して副次的

### パターン 4: マルチパネル (superfile)

独立した複数のディレクトリパネルを横に並べ、
各パネルが独自の状態を持つ。

コードレビューでの強み:

- 異なるディレクトリのファイルを比較可能
- 異なるワークフローに対応する柔軟なレイアウト

コードレビューでの弱み:

- Read-only ビューアには複雑すぎる
- パネル管理が認知負荷を増加

## コードレビュー TUI ビューアへの推奨事項

### レイアウトアプローチ

gracilius (read-only コードレビュー TUI) への推奨は
複数パターンの要素を組み合わせたもの:

1. 主レイアウトは2ペイン:
   左にファイルツリー、右にエディタ/diff ビュー。
   gracilius の現在の実装と一致し、
   コードレビューワークフローに適合する。

2. lf/yazi からの比率採用:
   ペイン幅に設定可能な比率を使用。
   デフォルト `1:3` または `1:4` (tree:editor) で
   コードに十分なスペースを確保。
   lf のターミナル幅に応じた動的比率再設定の
   アプローチは採用する価値がある。

3. broot からのツリー:
   ファイルツリーペインには broot の剪定ツリービュー
   (関連パスのみ表示) がフラットなディレクトリ
   リストよりもコードレビューに有用。
   変更のあるファイルをハイライトし、
   未変更ファイルは折り畳み可能にすべき。

4. Parent 列は不要:
   Miller columns と異なり、コードレビュービューアは
   Parent ディレクトリ列を必要としない。
   ツリーペインが完全な階層コンテキストを提供する。

### 具体的な推奨事項

#### ファイルツリーペイン (左)

- 変更ファイルを目立つように表示
  (diff ステータスインジケータ: added, modified, deleted)
- 未変更ディレクトリの折り畳みサポート
- ツリー内のフィルタ/検索 (broot パターン) を検討
- 幅: ターミナル幅の約 20-25%

#### エディタ / Diff ペイン (右)

- シンタックスハイライト付き diff ビュー (実装済み)
- 行番号 (old と new)
- vim キーバインドでスクロール可能
- 幅: ターミナル幅の約 75-80%

#### ステータス / 情報エリア

- lf の ruler コンセプトを採用:
  現在のファイル情報、レビュー進捗、接続状態を
  下部1行に表示
- ミニマルに保つ (1-2行)

#### マウス対応

- ツリーでファイル選択のクリック (実装済み)
- ペインボーダーのドラッグリサイズ (実装済み)
- 両ペインの独立スクロール

#### キーボードナビゲーション

- 両ペインで Vi ライクキー
- Tab または Ctrl+h/l でペイン間のフォーカス切り替え
- `/` でエディタ内検索
- `n`/`N` で次/前の変更ファイルへ移動

#### レスポンシブ動作

- 狭いターミナル (<80列):
  ツリーペインを非表示にしてオーバーレイで表示、
  またはツリーとエディタが交互に表示される
  single-pane モードへ切り替えを検討
- lf の動的比率調整パターンに従う

### Read-Only ビューアに有効なパターン

これらのファイルマネージャの調査から、
read-only コードレビューに最も適用可能なパターン:

1. 最小限のコントロール:
   ファイル操作 UI (copy/move/delete) が不要なため
   コンテンツ表示により多くのスペースを確保できる

2. プレビュー品質への注力:
   シンタックスハイライト、diff カラーリング、
   行番号が不可欠

3. 高速ナビゲーション:
   vim キー、検索、フィルタリングが
   マウス操作より重要

4. ステータスコンテキスト:
   "3 of 12 files reviewed" の表示が
   ファイルパーミッション表示より価値がある

5. デバウンスされた選択:
   lf と yazi は選択変更をデバウンスして
   不要な更新を回避する。
   選択がネットワーク通信をトリガーする
   gracilius の MCP プロトコルでは特に重要

### Miller Columns vs 2ペイン (コードナビゲーション)

Miller columns (3列) は
ディレクトリ階層のトラバーサルに最適化されている。
2ペインレイアウト (tree + content) は
ファイルの内容確認に最適化されている。

コードレビューツールではファイル内容確認が主要ワークフロー:

1. 変更ファイルのリストを見る
2. 1つ選んで diff を表示
3. diff をレビューし、場合によりコメントを残す
4. 次の変更ファイルに移動

このワークフローは左ペインが永続的なナビゲーション補助、
右ペインが主要ワークスペースとなる2ペインレイアウトに
適合する。Miller columns は
比例的なメリットなしに不要な複雑さを追加する。

gracilius が現在使用している2ペインレイアウトは
このユースケースに対して正しいアーキテクチャ選択である。
