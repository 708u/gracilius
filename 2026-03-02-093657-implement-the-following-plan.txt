❯ vscode vim extensionはどうやってこれを解消している？ 

⏺ VS Code Vim extension は Quick Open (Cmd+P) をネイティブ UI 
  コンテキストとして扱い、vim bindings を適用しません。Quick Open
  が開いた時点で自動的に Insert Mode 相当になり:

  - 全 printable keys → filter 入力
  - Ctrl+N / Ctrl+P / 矢印 → リスト移動
  - Enter → 選択
  - Escape → 閉じる

  これが成立する理由は、VS Code が「エディタバッファ」と「ネイティブ UI
  ダイアログ」を明確に区別しているからです。Vim extension は editor context
  でのみ normal mode を適用し、Quick Open/Command Palette 等の UI
  要素には介入しません。

  gracilius でも同じ設計が可能です。search overlay
  は独立したモーダルなので、overlay が active な間は vim 的な keybinding
  を一時停止し、上記の keybinding に切り替えれば良いだけです。現在の実装でも
  m.search.active で分岐しているので、構造的には既にそうなっています。

❯ vimの検索の場合はどうなっている？ 

⏺ Vim の / 検索はそもそもこの問題が発生しない設計です。

  / を押すと command-line mode に入り:
  - 全 printable keys → 検索パターン入力
  - Enter → 検索確定、normal mode に戻る
  - n / N → normal mode で次/前のマッチに移動

  つまり「入力」と「結果のナビゲーション」が完全に別フェーズです。リアルタイムで
  結果リストを表示して選択する UI ではないので、衝突が起きません。

  これは bubbles/list の built-in filter と同じ2フェーズ設計です。

  ライブフィルタ型の picker を提供する Vim プラグインの場合:

  ┌────────────────┬─────────────────────────────────────────────────────────┐
  │     Plugin     │                 入力中のナビゲーション                  │
  ├────────────────┼─────────────────────────────────────────────────────────┤
  │ fzf.vim        │ Ctrl+N/Ctrl+P, 矢印                                     │
  ├────────────────┼─────────────────────────────────────────────────────────┤
  │ telescope.nvim │ Insert mode: Ctrl+N/Ctrl+P, 矢印。Esc で normal mode    │
  │                │ に入ると j/k                                            │
  ├────────────────┼─────────────────────────────────────────────────────────┤
  │ ctrlp.vim      │ Ctrl+J/Ctrl+K, 矢印                                     │
  └────────────────┴─────────────────────────────────────────────────────────┘

  全て「入力中は Ctrl 修飾 or
  矢印でナビゲーション」で統一されています。telescope だけは Esc で picker 内
  normal mode に切り替えて j/k を使える二段構えです。

  gracilius の search overlay は fzf/ctrlp 方式（矢印 +
  Ctrl+N/Ctrl+P）が妥当だと思います。
